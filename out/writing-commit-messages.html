How to write a conventional commit with scope

In this tutorial, we are going to discuss commit messages. Commit messages are a very imporant part of the git workflow. At its core, a commit message is a set of instructions for how the code should change from a previous state to a new state. When you push your code to your repository, you will be able to see your commits in the commit history. If your commits were written properly, then the commit history will tell a story of how the project progressed over time; and as a result, other developers will be able to trace the projects progression, and easily find and fix bugs.

How are commit messages written ?

There a few different formats for writing commit messages. Today we will focus on conventional commits . More specifically, we will deconstruct the process of writing a commit message with scope.

Here is what a commit message with scope looks like:

type(scope): subject

When we break down this format, we notice that it is made of three parts (type, scope, subject). Each of these parts has a specific definition. We will now break down each part individually.

What does type mean ?

Type is a tag that identifies the type of change that you are making.

These are some of the well-known commit types:

  • Feat – identifies a new feature.
  • Fix – identifies that a bug has been fixed.
  • Docs – identifies updates to the documentation such as the readme file, or the code comments.
  • Style – identifies updates to the css styles, code formatting, missing semi-colons, etc.
  • Refactor – identifies updates involving code refactoring, such as renaming files, moving functions above objects in a file, moving a file into a folder.
  • Chore – identifies changes such as adding / removing / updating assets, dependencies etc.
  • Test – identifies updates about tests.
  • Perf – identifies updates about performance improvements.
  • Build – identifies updates that affect the build system or external dependencies.
  • Revert – reverts a previous commit.

  • Content - identifies content updates, such as adding or modifying texts and images etc.
  • Enhance - identifies additions and updates to an already existing feature.

    • What does scope mean ?

    The scope identifies the area (file / files / specific parts) in the code where something has changed.

    What does subject mean ?

    Subject refers to a sentence that’s usually 50 characters long that describes the changes that you've made, and why you’ve made these changes.

    Rules when writing the subject:

  • The first word in your commit should identify an action. Example: add, create, document, modify, update, remove etc.
  • Write the commit in present-tense, imperative-style.
  • Write it as if you are giving orders to a person on how to change the code, or to the codebase on how it should change its behaviour.
  • Remove unnecessary punctuation marks.
  • Do not end the subject line with a period.

    • Final words and commit examples

    Now that you have an overall understanding of what the terms type, scope, and subject mean, we have one more key point to mention. When you are writing conventional commits, the whole message should be in lowercase. With that said, here are a few examples of conventional commits with scope:

    git commit -m "feat(nav): add a new feature to the nav component"
    git commit -m "fix(header): fix a useEffect bug in the header"
    git commit -m "doc(readme): add a component hierarchy section to the readme file"
    git commit -m "style(font): change font type from ubuntu to roboto"

    Linking issues to commits

    Imagine that you're working on a task that's listed in a Github issue. This issue has the number 17. Now, if you want to link this issue to your commit mesage, then at the end of the commit message, insert a hashtag + the issue number like this:

    git commit -m "fix(header): fix a useEffect bug in the header #17"
